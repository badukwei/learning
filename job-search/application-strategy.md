# 求職投遞策略

更新：2026-05-04

## 核心結論

現階段不走純海投，也不走超精細少量投遞。

採用策略：

**A 精投 + B 半客製化大量投遞 + C 快速補量**

目標是在離職前先建立穩定的面試 pipeline，而不是把每一份履歷打磨到完美。

## 為什麼不是純海投

- 容易把個人優勢稀釋掉
- 很多職缺其實不適合，快速被拒不代表市場不要你
- 只按 apply，沒有額外訊號，回覆率通常偏低

## 為什麼不是純精投

- 投遞速度太慢
- 在離職前需要面試機會，不適合一週只投 2-3 份
- 太容易卡在過度分析，沒有實際累積市場回饋

## 職缺分級

### A 級職缺

條件：

- 真的想去
- 與背景高度符合
- 產品、薪資、遠端條件都合理

做法：

- 客製化履歷 summary
- 調整最前面的 2-3 條經驗 bullet
- 補 1-2 句為什麼適合這家公司
- 如果可能，搭配 LinkedIn 接觸

建議量：

- 每週 2-3 個

### B 級職缺

條件：

- 大方向符合
- 屬於 Frontend / Product / DevTools / AI / SaaS / Fintech 範圍
- 可作為主要投遞池

做法：

- 使用主履歷
- 只調整 summary 與最相關經驗 bullet
- 每份控制在 15-25 分鐘

建議量：

- 每週 5-8 個

### C 級職缺

條件：

- 沒那麼理想，但可增加 pipeline 或作為面試練習

做法：

- 快速投遞
- 不做深度客製化

建議量：

- 每週 3-5 個

## 現階段判斷

目前狀況：

- 投了沒有回覆，或很快被拒

優先假設不是能力不夠，而是：

1. 履歷定位沒有快速打中
2. 投遞池太雜，混入過多不適合的職缺
3. 投遞方式只有 apply，沒有額外訊號

## 履歷應該主打的賣點

- Frontend Engineer with SDK experience
- Built security monitoring for a live DeFi product
- Strong TypeScript / React / product delivery background
- Comfortable with AI workflow and developer tooling

不要把自己包裝成「普通前端工程師」。

## 接下來的投遞原則

- 在職期間：每週 8-12 投
- 離職後：每週 15-25 投
- 時間分配：70% B 級，20% A 級，10% C 級

## 這週優先任務

1. 改一版履歷主 summary
2. 回頭檢查最近 10 個投遞，分類失敗原因
3. 新投 5-6 個更精準職缺
4. 至少 2-3 個投遞搭配 LinkedIn 訊息或 coffee chat 式接觸

## 追蹤指標

- 10-15 投後是否出現 recruiter screen
- 被拒原因是否集中在地區 / seniority / 技能落差
- LinkedIn 接觸是否有回覆
- 哪類職缺最容易有正向訊號
