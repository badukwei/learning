# Claude Code — Context 管理學習筆記

---

## 為什麼 Context 很重要

Claude Code 每次對話都從空白開始，不記得上一次說了什麼。而且語言模型本身只能處理文字，沒辦法自己去讀檔案或跑指令——這些都要靠 tool use 系統來完成。

Context 的品質直接決定 Claude 的輸出品質：

- **給太少**：Claude 不知道你的專案架構，亂猜、問你一堆問題
- **給太多**：無關的雜訊讓 Claude 分心，反而效果變差
- **剛剛好**：Claude 拿到它需要的資訊，直接做事

---

## CLAUDE.md — 持久記憶

CLAUDE.md 是 Claude Code 的「常駐記憶」，每次對話開始時會自動載入。你不需要每次重新解釋專案是什麼、規範是什麼。

### 三種 CLAUDE.md 位置

| 類型 | 位置 | 說明 |
|------|------|------|
| **Machine（全域）** | `~/.claude/CLAUDE.md` | 對你機器上所有專案都生效，放個人偏好 |
| **Project（專案）** | `專案根目錄/CLAUDE.md` | 提交進 git，整個團隊共用，放專案規範 |
| **Local（個人覆蓋）** | `專案根目錄/CLAUDE.local.md` | 不提交，只對你自己生效，放個人對這專案的偏好 |

**優先順序：Local > Project > Machine**

同樣的指令在多個層級出現時，越具體的越優先。例如全域設定 2 格縮排，專案設定 4 格縮排，那這個專案就用 4 格。

### CLAUDE.md 適合放什麼

- 專案架構說明（目錄結構、主要模組）
- Coding 規範（TypeScript strict、named exports 等）
- 常用指令（如何跑測試、如何啟動 dev server）
- 重要注意事項（哪個檔案不能直接改、有什麼特殊邏輯）
- 關鍵檔案引用（例如 `@src/db/schema.ts` 讓 Claude 每次都帶著 schema）

## 指令一覽

### `/init`
在專案根目錄執行，Claude 會掃描整個 codebase，自動產出 `CLAUDE.md`。

```bash
# 在專案根目錄執行
/init
```

**注意：**
- 第一次用新專案時執行
- 產出的是草稿，建議手動刪掉不需要的部分（context 越精簡越好）
- 之後持續用 `#` 來累積新規則

---

### `#`（Memory Mode）
在 Claude Code 裡輸入 `#` 開頭，用自然語言直接編輯 CLAUDE.md，不需要手動開檔案。

```
# 之後不要用 var，只用 const 或 let
# commit message 一律用英文
# 改動超過 20 行的東西前先問我
```

Claude 會把這個規則智慧地寫進對應的 CLAUDE.md 層級。

**最佳使用時機：**
- Claude 剛犯了某個錯誤，馬上 `Escape` 中斷，然後 `#` 加規則
- 這樣之後就不會再犯同樣的錯

---

### `@`（指定檔案）
在請求裡用 `@` 點名特定檔案，把它加進這次請求的 context。

```
@src/db/schema.ts 根據這個 schema 幫我寫一個新的 query
@src/api/user.ts 這個檔案有個 bug，幫我找
```

**為什麼要用：**
- 不用 `@` 的話，Claude 會自己搜尋相關檔案，但不保證找到最重要的那個
- 用 `@` 可以確保關鍵檔案一定在 context 裡，而且不會帶入一堆無關的東西

**最佳實踐：**
把最常需要的關鍵檔案（例如 database schema、型別定義）直接引用在 `CLAUDE.md` 裡，這樣每次請求都自動帶入，不需要每次手動 `@`。

## Context 控制指令

光是建立 context 還不夠，在長時間的對話中，context 會逐漸累積雜訊，這時需要主動管理。

### `Escape`（中斷）
按一次 Escape，停止 Claude 目前的輸出，重新下指令。

**使用時機：**
- Claude 跑偏了，往錯誤方向執行
- 想改變請求方向

---

### `Escape + #`（中斷並記憶）
中斷之後馬上加規則，防止 Claude 重複犯同樣的錯誤。

```
# 不要直接刪除檔案，要先問我
```

這是最強的錯誤防止機制，當下抓到問題馬上記住，後續不會再發生。

---

### `Double Escape`（時光倒流）
按兩次 Escape，顯示所有對話紀錄，可以跳回之前某個節點重新開始。

**使用時機：**
- 走錯方向，debug 了一堆沒用的東西
- 想回到做出錯誤決定之前的狀態
- 保留 Claude 之前學到的有用知識，但跳過中間那段無效的來回

---

### `/compact`（壓縮對話）
把整個對話歷史摘要壓縮，但保留 Claude 在這次對話中學到的知識。

**使用時機：**
- 對話很長，開始有很多不相關的 debug 紀錄
- 但 Claude 已經對這個任務有足夠的理解，不想從頭開始

**vs `/clear` 的差別：**
- `/compact` = 保留知識，清掉雜訊
- `/clear` = 完全清空，全部重來

---

### `/clear`（完全清空）
刪除整個對話歷史，完全從新開始。

**使用時機：**
- 換完全不同的任務
- 目前的 context 已經太亂，重新來比較快

---

## 總結：什麼情況用什麼

| 情況 | 用什麼 |
|------|--------|
| 新專案第一次用 | `/init` |
| Claude 犯了某個錯，要它記住 | `Escape` + `#` |
| 想帶入特定檔案 | `@檔案路徑` |
| Claude 跑偏，要重新下指令 | `Escape` |
| 對話太長太亂，但 Claude 已有背景知識 | `/compact` |
| 完全換任務 | `/clear` |
| 走錯方向，想回到之前的節點 | `Double Escape` |
| 加全域規則 | `~/.claude/CLAUDE.md` |
| 加專案規則（共用） | `專案根目錄/CLAUDE.md` |
| 加個人規則（不共用） | `專案根目錄/CLAUDE.local.md` |

---

## 核心原則

> **讓 Claude 在每次請求時剛好拿到它需要的資訊，不多也不少。**

Context 不是越多越好。每一行 CLAUDE.md 都在跟你的實際任務競爭注意力。精簡、具體、相關——這是 context 管理的核心。
