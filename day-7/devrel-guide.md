## 1. DevRel 是什麼工作？

> "DevRel serves as a middle ground, connecting marketing messaging with developers through content creation and technical expertise."
> — Ethan Francis, Chainstack DevRel Lead

DevRel（Developer Relations）是技術與商業的橋梁。不是純工程師，也不是傳統行銷。

### 三個核心方向

| 方向 | 說明 | 比重（一般）|
|------|------|------------|
| **Content**（內容）| 教學文章、tutorial、demo、影片、documentation | ~40% |
| **Community**（社群）| Discord 回答、Twitter/X 互動、Forum 討論、Meetup | ~35% |
| **Product Feedback**（產品回饋）| 收集開發者痛點、傳回給工程 / PM 團隊 | ~25% |

### Web3 DevRel 的三階段漏斗（Daria Volkova 框架）

```
Awareness（讓人知道）
  → 演講、meetup、social media 貼文、Twitter Space

Trust（建立信任）
  → 工作坊、AMA、技術深度文章、開源貢獻

Conversion（讓人用）
  → 文件改善、integration guide、hackathon sponsorship
```

## 2. 真實的一天（Christina Lin, Redpanda DevRel）

這是遠端 DevRel 的真實工作日樣貌：

```
早上
  → 瀏覽 social media、Discord / Slack 社群頻道
  → 找「開發者在卡什麼問題」
  → 記錄模式：是文件缺口？常見誤解？沒有的功能？

上午
  → 把發現傳給內部（工程、PM、行銷）
  → 討論這週要做哪些內容

下午
  → 寫 blog / tutorial / 準備工作坊內容
  → 可能是影片腳本、可能是 demo code

不定期（每週）
  → 準備 conference talk
  → 回 GitHub issue / PR
  → 出現在開發者活動
```

**關鍵洞察**：寫 code 只佔 10–25%，大多數時間是寫作、溝通、和人互動。

## 3. DevRel 必備的 4 個能力

來源：Sam Julien（前 Auth0 DevRel）的整理，被業界廣泛引用。

| 能力 | 具體表現 | Wei-Chun 現況 |
|------|---------|--------------|
| **Empathy（同理心）** | 理解開發者的挫敗感，站在使用者角度設計內容 | ✅ 有 DeFi 用戶設計經驗 |
| **Love of learning** | 快速上手新技術，不害怕「還不會」 | ✅ AI 工具、新鏈快速學 |
| **Project management** | 同時跑 5-10 個任務（文章、demo、conference、社群） | ⚠️ 需刻意練習，容易失焦 |
| **Communication** | 能讓複雜的技術概念讓初學者也懂 | ✅ 中英文雙語，有教學傾向 |

## 4. Web3 DevRel 的特殊要求

Web3 和 Web2 DevRel 有幾個明顯差異：

| 面向 | Web2 DevRel | Web3 DevRel |
|------|------------|------------|
| 社群文化 | 較企業化、正式 | 更去中心化、Twitter Space 重要 |
| 評估標準 | 你用過這個工具嗎 | **你在這條鏈上做過什麼？** |
| 稀缺性 | 競爭者多 | Sui/Move 稀缺，空間更大 |
| 內容平台 | blog、YouTube | Twitter/X、Farcaster、dev.to + Discord |
| 入門門檻 | 有溝通能力即可 | 需要真實的 on-chain 使用 / 貢獻記錄 |

> **核心差異**：Web3 社群很小，每個貢獻都被看見。
> 在 Sui Developer Forum 回答一個問題，Mysten Labs 的人很可能會看到。

## 5. 入行路徑：從工程師到 DevRel（Step-by-Step）

整合 Sam Julien、developerrelations.com、Chainstack 的共同建議：

### Phase 1：在現有角色中累積 DevRel 素材（= 現在的 Wei-Chun）

> 不需要換工作——DevRel 的技能可以在現職開始練。

- [ ] **開始在公司內部做技術分享**（演講、教學、文件）
- [ ] **把做過的東西寫成文章**——就算只是技術記錄，也比沒有強
- [ ] **用英文寫**——面向全球 Web3 社群，中文侷限了受眾

### Phase 2：建立公開可見度（Month 1-2 的重點）

> Hiring managers 看的是「你在公開場合做了什麼」，不只是履歷。

```
GitHub
  → demo repo（不只是 fork，要自己寫的）
  → open source 貢獻（PR 記錄）
  → README 要好看、說清楚這個 demo 解決什麼問題

dev.to / Medium
  → 第一篇：「我用 XXX 做了什麼，這是我踩到的坑」
  → 比「XXX 是什麼入門介紹」更有價值

Twitter/X
  → 分享學習過程（建在公開場合）
  → 轉發文章時加一句自己的看法

Discord / Forum
  → 在 Sui Developer Forum 回答問題
  → 不需要每次都完整回答，「這裡有一個 example 可以參考」就夠了
```

### Phase 3：投遞時的 Portfolio 結構

DevRel 面試要帶的「portfolio」，不是 PDF，是一組連結：

```
GitHub profile
  ├── demo repo（Holepunch / Walrus / Move 至少一個）
  └── 對外 open source 貢獻紀錄

dev.to / Medium
  └── 至少 2-3 篇英文技術文章

Twitter/X
  └── 有在討論技術的帳號（不需要很多粉絲）

選擇性加分
  └── 演講錄影 / 工作坊紀錄
  └── Sui Developer Forum 的回答記錄
```

## 6. 關鍵心態轉換

### 「學了才寫」→ 「邊學邊寫」

大多數入行者都犯一個錯：等技術學夠了才開始公開。

> "Just turn your camera on and work for people."
> — Austin Griffith

**學習 Holepunch 的第一天就開始記錄踩坑過程。** 這比學完後回頭寫的文章更真實、更有共鳴。

### 「全都要會」→ 「選一個深入」

> "I see that there is no need to be excellent in everything."
> — Daria Volkova, Web3 DevRel

Sui / Move 就是 Wei-Chun 的深度。不需要同時成為 EVM / Solana / Cosmos 專家。

### 「等機會」→ 「先做起來」

developerrelations.com 最核心的建議：

> "Start contributing and building visibility in developer communities immediately, rather than waiting for a formal job opening."

在社群露面 → 認識人 → 機會找上你，比投履歷有效得多。

## 8. 常見誤解

| 誤解 | 真相 |
|------|------|
| 「DevRel 就是去演講」| 演講是 DevRel 的一小部分，更多是寫作和社群 |
| 「需要 7 年經驗」| 高門檻職缺如此，Tether 只要 3 年，很多職缺更低 |
| 「需要有大量粉絲才能投」| 品質比數量重要，2 篇有深度的文章勝過 100 篇轉發 |
| 「DevRel 不需要真的會 code」| Web3 DevRel 一定要會，開發者看得出來 |
| 「DevRel 就是行銷換個名字」| 不對，DevRel 是替開發者說話，必要時對公司說「不，這個 API 設計很差」|
